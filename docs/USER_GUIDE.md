## Работа с продуктами

### Список продуктов (главная страница)

На главной странице отображаются карточки всех продуктов. Каждая карточка содержит:
- Название продукта
- Описание (если заполнено)
- Стек технологий
- Ответственный
- Количество открытых задач
- Статус (active / archived)

### Создание продукта

1. Нажмите кнопку **«+ Продукт»** в правом верхнем углу
2. Заполните форму:
   - **Название** (обязательное) — название продукта
   - **Описание** — краткое описание назначения
   - **Репозиторий (URL)** — ссылка на GitLab/GitHub
   - **Стек технологий** — основные технологии (Node.js, Vue, PostgreSQL...)
   - **Ответственный** — имя ответственного за продукт
   - **Путь к проекту** — путь к директории проекта на сервере
   - **ID системы (Rivc.Connect)** — числовой ID системы в HelpDesk Rivc.Connect (необязательно)
   - **ID модуля (Rivc.Connect)** — числовой ID модуля в HelpDesk Rivc.Connect (необязательно)
3. Нажмите **«Сохранить»**

### Редактирование продукта

1. На карточке продукта нажмите **«Редактировать»**
2. Измените нужные поля
3. Нажмите **«Сохранить»**

### Архивирование / активация

- **«Архивировать»** — переводит продукт в статус `archived`
- **«Активировать»** — возвращает продукт в статус `active`

### Удаление продукта

1. На карточке нажмите **«Удалить»**
2. Подтвердите удаление в диалоге

> Каскадное удаление: все задачи и релизы продукта будут удалены.

## Работа с задачами (Issues)

Задачи отображаются на странице продукта в разделе **«Задачи»**.

### Фильтрация

Используйте кнопки фильтра над таблицей:
- **Все** — все задачи
- **Open** — только открытые
- **In Release** — включённые в релиз
- **Done** — завершённые
- **Closed** — закрытые

### Создание задачи

1. Нажмите **«+ Задача»**
2. Заполните форму:
   - **Название** (обязательное) — краткое описание проблемы или улучшения
   - **Описание** — подробное описание
   - **Тип** — `bug` / `improvement` / `feature`
   - **Приоритет** — `critical` / `high` / `medium` / `low`
3. Нажмите **«Сохранить»**

### Редактирование задачи

1. В строке задачи нажмите **«Ред.»**
2. Измените поля
3. Нажмите **«Сохранить»**

### Удаление задачи

1. В строке задачи нажмите **«Уд.»**
2. Подтвердите удаление

### Статусы задач

| Статус | Описание |
|--------|----------|
| `open` | Задача создана, ожидает включения в релиз |
| `in_release` | Задача включена в релиз (устанавливается автоматически) |
| `done` | Задача завершена (устанавливается при публикации релиза) |
| `closed` | Задача закрыта вручную |

## Работа с релизами

Релизы отображаются на странице продукта в разделе **«Релизы»**.

### Создание релиза

1. Нажмите **«+ Релиз»**
2. Заполните форму:
   - **Версия** (обязательное) — номер версии (например, `1.0.0`, `v2.1`)
   - **Название** (обязательное) — человекочитаемое название релиза
   - **Описание** — release notes
   - **Включить задачи** — отметьте чекбоксами открытые задачи для включения в релиз
3. Нажмите **«Создать релиз»**

> При создании релиза выбранные задачи автоматически переходят в статус `in_release`. Система показывает уведомление с количеством затронутых задач.

### Просмотр задач релиза

Нажмите **«Показать задачи»** на карточке релиза, чтобы развернуть список включённых задач.

### Публикация релиза

1. На карточке релиза со статусом `draft` нажмите **«Опубликовать»**
2. Результат:
   - Статус релиза → `released`
   - Фиксируется дата выпуска
   - Все задачи релиза → `done`
3. Отображается уведомление с деталями изменений (количество задач → done)

### Удаление релиза

1. На карточке релиза нажмите **«Удалить»**
2. Подтвердите удаление
3. Отображается уведомление с количеством задач, возвращённых в `open`

> При удалении релиза задачи возвращаются в статус `open`.

### Подготовка пресс-релиза

Для **опубликованных** релизов доступна генерация PR-материалов.

1. На карточке опубликованного релиза нажмите **«Пресс-релиз»**
2. В модале настроек выберите:
   - **Каналы**: Соцсети (ВК, Telegram), Сайт, Битрикс24, СМИ — можно выбрать несколько
   - **Тональность**: Официальная / Дружелюбная / Техническая / Маркетинговая
   - **Аудитория**: Сотрудники / Клиенты / Техсообщество / Широкая аудитория
   - **Промпты для изображений**: генерировать описания для AI-генерации картинок
   - **Ключевые акценты**: на что обратить особое внимание (необязательно)
   - **Модель ИИ** и **Таймаут**
3. Нажмите **«Запустить»** — процесс появится в таблице процессов
4. После завершения кнопка сменится на **«Открыть пресс-релиз»**
5. В модале просмотра — табы по каналам, кнопки **«Скопировать канал»** и **«Скачать .md»**

### Статусы релизов

| Статус | Описание |
|--------|----------|
| `draft` | Черновик — формируется состав |
| `in_progress` | В работе |
| `released` | Опубликован — задачи завершены |

## Разработка релиза (develop_release)

Claude Code автоматически реализует все задачи релиза: создаёт ветку, пишет код, тесты, коммитит и пушит.

### Прогресс в реальном времени

Во время работы процесса `develop_release` в модале деталей отображается **визуальный степпер** с 7 контрольными точками:

| Фаза | Описание |
|------|----------|
| Подготовка репозитория | git checkout / git pull |
| Изучение кодовой базы | Чтение файлов, поиск по коду |
| Реализация задач | Создание и редактирование файлов |
| Написание тестов | Создание файлов `*.test.*` / `*.spec.*` |
| Запуск тестов | Выполнение тест-команды |
| Обновление документации | Изменения в `docs/` |
| Коммит и push | git commit + git push |

Контрольные точки определяются автоматически по действиям Claude Code (какие инструменты вызывает, с какими файлами работает). Пройденные фазы отмечаются зелёным, текущая — с анимацией пульсации, будущие — серым.

Степпер обновляется при каждом поллинге (каждые 4 секунды).

## Модели ИИ

Страница **«Модели ИИ»** (`/models.html`) содержит реестр AI-моделей, используемых для генерации задач.

### Добавление модели

1. Нажмите **«+ Модель»**
2. Заполните форму:
   - **Название** (обязательное) — человекочитаемое имя
   - **Model ID** (обязательное) — идентификатор модели у провайдера (например, `qwen3-coder:30b`, `claude-sonnet-4-6`)
   - **Провайдер** — Ollama / MLX / OpenAI / Anthropic / Google
   - **Тип развёртывания** — Локальная / Облачная
   - **API Key** — ключ для облачных провайдеров (маскируется при отображении)
   - Описание, размер параметров, длина контекста — опционально
3. Нажмите **«Создать»**

### Автообнаружение локальных моделей

При выборе типа «Локальная» появляется секция **«Обнаружить установленные»**:

1. Нажмите **«Обнаружить установленные»**
2. Система проверит Ollama (порт 11434) и MLX (порт 8080)
3. Выберите модель из выпадающего списка — поля заполнятся автоматически

### Загрузка в GPU (Warmup)

Для локальных моделей доступна кнопка **«Загрузить»** — подготавливает модель в GPU для быстрого ответа.

### Фильтрация

Используйте выпадающий список для фильтрации: **Все типы** / **Локальные** / **Облачные**.

## Улучшение продукта (AI-генерация задач)

Функция позволяет с помощью ИИ сгенерировать задачи на улучшение продукта.

### Запуск

1. Перейдите на страницу продукта
2. Нажмите кнопку **«Улучшение продукта»** рядом с «Редактировать»

### Фаза 1: Настройка

1. **Шаблон промпта** — выберите один из 6 встроенных шаблонов (Общие улучшения, UI, Производительность, Безопасность, Анализ конкурентов, Developer Experience) или напишите свой промпт
2. **Промпт** — заполняется автоматически при выборе шаблона, можно отредактировать
3. **Модель ИИ** — выберите модель из реестра
4. **Количество задач** — от 1 до 10 (по умолчанию 5)
5. Нажмите **«Сгенерировать»**

### Фаза 2: Ожидание

Отображается спиннер. Генерация может занять до 2 минут в зависимости от модели.

### Фаза 3: Обзор предложений

1. Отображается список предложений ИИ — каждое с названием, типом, приоритетом и описанием
2. Отметьте чекбоксами нужные предложения (по умолчанию все выбраны)
3. Кнопки **«Выбрать все»** / **«Снять все»** для быстрого управления
4. Нажмите **«Создать выбранные (N)»** — выбранные предложения станут обычными задачами со статусом `open`
5. Кнопка **«Назад»** вернёт к форме для повторной генерации

## Очередь процессов

На странице **«Процессы»** (`/processes.html`) отображается виджет статистики очереди.

### Как работает очередь

- Каждый AI-провайдер имеет лимит параллельных процессов: ollama:1, mlx:1, claude-code:2, anthropic:3, openai:3, google:3
- Если все слоты заняты, новый процесс встаёт в очередь со статусом `queued`
- При завершении текущего процесса автоматически запускается следующий из очереди
- Queued-процесс можно отменить кнопкой **«Отменить»**

### Приоритеты

- **Normal** (0) — обычный приоритет
- **High** (1) — повышенный
- **Urgent** (2) — срочный (запускается первым)

## Планы (цепочки AI-процессов)

Страница **«Планы»** (`/plans.html`) содержит планы — цепочки AI-процессов с зависимостями.

### Создание плана

1. Нажмите **«+ План»**
2. Заполните: название, продукт, описание, поведение при ошибке (stop/skip)
3. Добавьте шаги: для каждого — модель, тип процесса, промпт, таймаут
4. Настройте зависимости: шаг может зависеть от завершения других шагов

### Запуск плана

1. Нажмите **«Запустить»** — план переходит в статус `active`
2. Планировщик (каждые 30 сек) находит готовые шаги и запускает их через очередь
3. Прогресс отображается в прогресс-баре

### Статусы плана

| Статус | Описание |
|--------|----------|
| `draft` | Черновик — можно редактировать шаги |
| `scheduled` | Запланирован на определённую дату |
| `active` | Выполняется — шаги запускаются автоматически |
| `completed` | Все шаги завершены |
| `failed` | Ошибка в шаге (при on_failure=stop) |
| `cancelled` | Отменён вручную |

## Тикеты Rivc.Connect (RC)

Для продуктов, привязанных к системе/модулю Rivc.Connect, доступна синхронизация тикетов HelpDesk.

### Синхронизация тикетов

1. Откройте страницу продукта с настроенным RC (ID системы и модуля)
2. Перейдите на вкладку **«Тикеты RC»**
3. Нажмите **«Синхронизировать»** — загружает все незакрытые тикеты из MS SQL в локальный кэш

### Фильтрация тикетов

- **Все** — все кэшированные тикеты
- **Новые** — ещё не обработаны
- **Импортированные** — уже созданы как задачи Kaizen
- **Игнорированные** — помечены как нерелевантные

### Импорт тикетов

1. Отметьте чекбоксами нужные тикеты (только новые)
2. Нажмите **«Импортировать выбранные»** — создаются задачи Kaizen с привязкой `rc_ticket_id`

Или откройте тикет и нажмите **«Импортировать»** для единичного импорта.

### Игнорирование тикетов

Отметьте тикеты → **«Игнорировать выбранные»**. Игнорированные тикеты не показываются в фильтре «Новые».

## AI-формирование релизов

AI анализирует открытые задачи и предлагает оптимальное распределение по релизам.

### Запуск

1. На странице продукта нажмите **«Сформировать релиз (AI)»** (кнопка видна при ≥ 2 открытых задачах)
2. В модале настроек выберите:
   - **Стратегия группировки**: Сбалансированная / Критичное первым / По темам / Всё в один
   - **Макс. релизов** (1–10)
   - **Модель ИИ**
   - **Таймаут**
   - **Авто-утверждение** — если включено, релизы создаются автоматически
3. Нажмите **«Запустить»**

### Ручной режим (по умолчанию)

1. После завершения AI-процесса открывается модал обзора
2. Для каждого предложенного релиза отображается: версия, название, описание, приоритет, список задач с чекбоксами, обоснование группировки
3. Можно снять чекбоксы с ненужных задач
4. Нажмите **«Утвердить и создать релизы»**

### Стратегии группировки

| Стратегия | Описание | Когда использовать |
|-----------|----------|--------------------|
| Сбалансированная | Равномерные релизы по объёму и приоритету | По умолчанию |
| Критичное первым | Критичные баги в первый релиз | Много багов |
| По темам | Группировка по функциональным областям | Разнородные задачи |
| Всё в один | Один релиз со всеми задачами | Мало задач (< 8) |

## MCP-сервер (управление через Claude Code)

Kaizen имеет встроенный MCP-сервер, позволяющий управлять системой через Claude Code.

### Подключение

MCP-сервер подключается автоматически через `~/.claude/settings.json`. Требуется запущенный Kaizen на порту 3034.

### Использование

В Claude Code доступны 29 инструментов с префиксом `kaizen_`. Примеры:

**Просмотр продуктов:**
> «Покажи список всех продуктов в Kaizen»

**Улучшение продукта:**
> «Запусти анализ безопасности продукта A-CDM через Kaizen»

**Полный конвейер:**
> «Запусти полный конвейер улучшения для BLST: шаблон general, 5 предложений, версия 1.1.0, утверди всё критичное»

### kaizen_run_pipeline — сквозной конвейер

Самый мощный инструмент. Выполняет полный цикл одной командой.

**Базовые этапы (всегда выполняются):**

1. Генерация AI-предложений по выбранному шаблону
2. Ожидание завершения (polling)
3. Автоматическое утверждение по правилам:
   - `all` — утвердить все
   - `high_and_critical` — только high и critical приоритет
   - `critical_only` — только critical
   - `none` — ничего не утверждать (ручной режим)
4. Создание релиза из утверждённых задач
5. Генерация спецификации

**Опциональные этапы (включаются параметрами):**

6. **Разработка** (`develop: { enabled: true }`) — Claude Code реализует задачи
7. **Публикация** (`develop: { auto_publish: true }`) — автоматически при успешных тестах
8. **Пресс-релиз** (`press_release: { enabled: true }`) — PR-материалы для каналов

## Автоматизация продуктов

Каждый продукт можно настроить на автоматическое выполнение конвейера без ручного вмешательства.

### Настройка

1. Откройте страницу продукта
2. Перейдите на таб **«Автоматизация»**
3. Настройте нужные секции и нажмите **«Сохранить настройки»**

### RC Auto-Sync

Автоматическая синхронизация тикетов из Rivc.Connect по расписанию.

- **Интервал** (часы): 1 = каждый час, 24 = раз в день, 168 = раз в неделю
- **Авто-импорт**: автоматический импорт тикетов с выбранными приоритетами (Critical, High, Medium)

### Auto-Pipeline

Автоматический запуск полного конвейера (improve → approve → release → spec → ...).

**Триггеры:**

| Триггер | Описание | Когда использовать |
|---------|----------|-------------------|
| threshold | При накоплении N открытых задач | Постоянный поток задач |
| schedule | По расписанию (каждые N часов) | Регулярные релизы |
| on_sync | После RC-синхронизации при новых тикетах | Быстрая реакция на баги |

**Настройки конвейера:**
- Модель ИИ, шаблон промпта, количество предложений
- Правило утверждения (All / High+Critical / Critical only)
- Стратегия версий: авто-инкремент (minor bump от последней)

**Дополнительные этапы:**
- Разработка (develop_release) с авто-публикацией
- Пресс-релиз с выбором каналов и тональности

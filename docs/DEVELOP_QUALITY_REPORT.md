# Отчёт: Анализ ошибок AI-разработки и план улучшений

> Дата: 2026-03-13
> Контекст: 4 релиза Rivc.Schedule (v0.7.0–v0.10.0), разработанные через Kaizen

## Что сломал Kaizen и почему

| Ошибка | Причина | Корень проблемы |
|--------|---------|-----------------|
| `SELECT Id` вместо `ID_USER` | AI придумал имена колонок | **Не читал схему БД** |
| `SELECT FIO` — нет такой колонки | AI додумал поле | **Не читал схему БД** |
| `PID, FID, Fname, UserID` | AI придумал имена FK | **Не читал схему БД** |
| `v-model:visible` на props (8 файлов) | AI не знает Vue 3 ограничения | **Не проверил билд** |
| Отсутствие NuGet-пакета | AI добавил `IHttpContextAccessor` без зависимости | **Не проверил билд** |
| `npm ci` сломался | AI изменил зависимости, не обновил lock | **Не проверил Docker-сборку** |

## Три системных дефекта

### 1. Промпт `develop_release` не указывает на документацию

В `prepare_spec` (строка 558) есть явная инструкция:
```
Начни с чтения CLAUDE.md и файлов из docs/ — там контекст проекта, архитектура, API, схема БД
```

А в `develop_release` (строка 1382) только общее:
```
Изучи структуру проекта, ключевые файлы, архитектурные паттерны.
```

AI не знает, что в CLAUDE.md есть точные имена колонок, а в docs/ — полная схема БД. Он **угадывает** вместо того, чтобы **читать**.

### 2. Нет валидации результата перед коммитом

Шаги 4-5 (тесты) в промпте есть, но:
- Нет шага `dotnet build` / `npm run build` — проверки компиляции
- Нет Docker-сборки как smoke-теста
- AI коммитит код, который не компилируется

### 3. Нет защиты существующего функционала

Каждый релиз разрабатывается в изоляции. AI не знает, что авторизация уже работает и её нельзя ломать. Нет списка "критических путей" которые должны оставаться рабочими.

## Список улучшений

### P0 — Правки промптов (реализовано 2026-03-13)

#### 1. Обязательные ссылки на документацию в промпте develop_release
Добавить в системный промпт блок с явным указанием читать CLAUDE.md и docs/.

#### 3. Pre-build шаг в промпте
Обязательная проверка компиляции (`dotnet build` / `npm run build`) перед коммитом.

#### 6. Baseline тесты
Запуск тестов ДО изменений — если проходили, после изменений тоже должны проходить.

### P1 — Конфигурация продукта (запланировано)

#### 2. Поле `context_files` в продукте
JSONB-поле со списком файлов/папок, которые AI обязан прочитать перед разработкой.

#### 5. Список "защищённых путей" (critical paths)
Критические файлы/модули с описанием — AI предупреждается о них в промпте.

#### 7. Спецификация должна ссылаться на конкретные файлы
Усилить формат spec: точные пути, фрагменты существующего кода, имена из документации.

### P2 — Новые типы шагов (запланировано)

#### 4. Новый тип шага `validate_build` в планах
Автоматическая проверка сборки после develop_release.

#### 8. Автоматический review после develop
AI-ревью git diff: проверка на придуманные имена, сломанные импорты, компиляцию.

## Предложения по улучшению системы (Claude Code Review, 2026-03-13)

> Глубокий анализ кодовой базы Kaizen v1.11.0 — безопасность, качество кода, производительность, тестирование, UX.

### Безопасность

| # | Проблема | Описание | Приоритет | Усилия |
|---|----------|----------|-----------|--------|
| S1 | **Аутентификация отсутствует** | Все 55+ эндпоинтов публичны. Любой в сети может удалить продукты/процессы. Добавить JWT или сессионную аутентификацию + RBAC (admin/product-owner/viewer) | **Критично** | 8ч |
| S2 | **Нет валидации входных данных** | POST/PUT эндпоинты проверяют только non-empty. Нет валидации типов, длин, форматов. Использовать zod или joi для schema validation на всех эндпоинтах | **Высокий** | 4ч |
| S3 | **Имя git-ветки не валидируется** | `config.git_branch` напрямую передаётся в `git checkout -b`. Специальные символы, `../` — потенциальная инъекция. Добавить regex: `/^[a-z0-9._/-]+$/i` | **Высокий** | 30м |
| S4 | **Нет rate limiting** | В `server/index.js` отсутствует middleware для ограничения запросов. Уязвимость к DoS. Добавить `express-rate-limit` (100 req/min для API, 10 req/min для AI-процессов) | **Высокий** | 1ч |
| S5 | **Scheduler auto-pipeline без ограничения частоты** | Если threshold=1, pipeline запускается бесконечно. Добавить per-product rate limiting (min_interval_hours между pipeline запусками) | **Средний** | 1ч |
| S6 | **Нет HTTPS и security headers** | Отсутствуют заголовки: HSTS, X-Content-Type-Options, X-Frame-Options, CSP. Добавить `helmet` middleware | **Средний** | 30м |

### Качество кода

| # | Проблема | Описание | Приоритет | Усилия |
|---|----------|----------|-----------|--------|
| Q1 | **Нет структурированного логирования** | Только `console.log/error`. Нет уровней, ротации, файлового вывода. Внедрить `pino` — быстрый JSON-логгер с уровнями и ротацией | **Высокий** | 3ч |
| Q2 | **QueueManager глотает ошибки** | `onProcessDone` обёрнут в пустой try/catch — баги в Scheduler не видны. Добавить логирование ошибок в catch-блок | **Высокий** | 15м |
| Q3 | **Inconsistent HTML escaping** | `escapeHtml()` доступен в `app.js`, но не все динамические innerHTML-вставки его используют. Аудит всех `innerHTML` на XSS | **Средний** | 2ч |
| Q4 | **process-runner.js — 1771 строк** | Монолитный файл. Разбить на модули: `run-improve.js`, `run-develop.js`, `run-tests.js`, `run-docs.js` — по типу процесса | **Средний** | 3ч |
| Q5 | **Нет graceful shutdown** | Scheduler/QueueManager не останавливаются при SIGTERM. Добавить обработчик `process.on('SIGTERM')` с остановкой таймеров и завершением текущих процессов | **Средний** | 1ч |

### Производительность

| # | Проблема | Описание | Приоритет | Усилия |
|---|----------|----------|-----------|--------|
| P1 | **Нет пагинации в API** | `processes.getAll()` возвращает ВСЕ записи. При 10,000 процессах — критическое замедление. Добавить `limit`, `offset` параметры + пагинацию в UI | **Высокий** | 3ч |
| P2 | **Polling вместо WebSocket/SSE** | Фронтенд опрашивает сервер каждые 4-10с. При 50 пользователях = 500 req/мин. Перейти на Server-Sent Events (SSE) для real-time статусов процессов | **Высокий** | 6ч |
| P3 | **Логи процессов растут бесконечно** | Таблица `kaizen_process_logs` не чистится. Добавить автоочистку: `DELETE WHERE created_at < NOW() - INTERVAL '90 days'` в Scheduler | **Средний** | 1ч |
| P4 | **Кэширование discover моделей** | Если Ollama/MLX недоступны, `/discover` ждёт таймаут 5с. Кэшировать результат на 5 минут | **Низкий** | 1ч |

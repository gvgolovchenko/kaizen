# Глубокий анализ Kaizen (改善) v1.6.0

> Дата анализа: 2026-03-08

---

## 1. Что уже построено

Kaizen — полноценная система непрерывного улучшения продуктов:

- **6 типов AI-процессов**: improve, prepare_spec, develop_release, roadmap_from_doc, prepare_press_release
- **6 AI-провайдеров**: ollama, mlx, claude-code, anthropic, openai, google
- **QueueManager** с контролем параллелизма по провайдерам
- **Scheduler** (тик 30с) с планами, шагами, зависимостями
- **50+ API эндпоинтов**, 9 таблиц, 7 страниц UI

По сути — это уже AI DevOps платформа.

## 3. Главная проблема: конвейер требует ручного управления

Сейчас, чтобы «прогнать» продукт через полный цикл улучшения, нужно:

1. Руками запустить "Улучшить" → дождаться → утвердить задачи
2. Руками создать релиз из утверждённых задач
3. Руками запустить "Подготовить спец." → дождаться
4. Руками запустить "Разработать" → дождаться
5. Руками запустить "Пресс-релиз" → дождаться
6. Руками опубликовать релиз

Даже с Plans — нужно вручную собрать план, шаги, запустить, потом утверждать результаты.

## 4. Два пути к непрерывному конвейеру

### Путь 1: MCP-сервер для Kaizen (рекомендуется первым)

**Зачем**: Позволяет Claude Code (и другим AI-агентам) управлять Kaizen как инструментом. Вместо UI — прямые вызовы.

```
Claude Code + Kaizen MCP = AI-агент, который сам управляет продуктами
```

#### Какие MCP-инструменты создать:

| Tool | Описание | Пример вызова |
|------|----------|---------------|
| `kaizen_list_products` | Список продуктов | — |
| `kaizen_get_product` | Детали продукта + открытые задачи | `{product_id}` |
| `kaizen_improve_product` | Запустить AI-улучшение | `{product_id, template, model_id}` |
| `kaizen_approve_suggestions` | Утвердить предложения | `{process_id, indices}` |
| `kaizen_create_release` | Создать релиз из задач | `{product_id, issue_ids, version}` |
| `kaizen_prepare_spec` | Сгенерировать спецификацию | `{release_id, model_id}` |
| `kaizen_develop_release` | Запустить разработку | `{release_id}` |
| `kaizen_publish_release` | Опубликовать релиз | `{release_id}` |
| `kaizen_get_process_status` | Статус процесса | `{process_id}` |
| `kaizen_run_pipeline` | Полный конвейер одной командой | `{product_id, model_id}` |

#### Архитектура MCP-сервера:

```
kaizen-mcp-server/
├── index.js              # MCP server (stdio transport)
├── tools/
│   ├── products.js       # CRUD tools
│   ├── pipeline.js       # run_pipeline — полный конвейер
│   └── processes.js      # Управление процессами
└── lib/
    └── api-client.js     # HTTP-клиент к Kaizen API (localhost:3034)
```

MCP просто вызывает существующий REST API Kaizen — ничего не дублируется.

#### Результат:

Ты говоришь Claude Code:
> «Улучши продукт A-CDM, используя шаблон security, утверди всё критичное, собери релиз 2.1.0 и подготовь спецификацию»

И он это делает через MCP-инструменты.

#### Почему MCP первый:

1. **Быстро** — MCP-сервер это ~300 строк, оборачивает существующий API
2. **Гибко** — Claude Code может комбинировать инструменты как угодно
3. **Не нужен autopipeline** — Claude Code сам и есть autopipeline, только умнее (он может принимать решения об утверждении на основе контекста)
4. **Интеграция** — сразу работает в Claude Code workflow
5. **Тестирование** — прогнал вручную через Claude Code, убедился что работает, потом автоматизировал через Plans/Cron

### Путь 2: Автопайплайн в самом Kaizen (autopipeline)

Добавить новую сущность — **Pipeline** (конвейер), которая автоматизирует полный цикл:

```
Pipeline = Plan + автоматическое утверждение + триггеры
```

#### Отличие от Plans:

| | Plans (сейчас) | Pipeline (новое) |
|---|---|---|
| Утверждение задач | Вручную | Автоматическое по правилам |
| Создание релиза | Вручную | Автоматическое |
| Триггер запуска | Ручной / по расписанию | Cron / webhook / событие |
| Цикличность | Одноразовый | Повторяющийся |

#### Минимальная реализация:

Новая таблица `kaizen_pipelines`:

```sql
id UUID PRIMARY KEY,
product_id UUID REFERENCES kaizen_products(id),
name TEXT NOT NULL,
cron_expression TEXT,              -- '0 9 * * 1' (каждый понедельник в 9:00)
auto_approve_rules JSONB,          -- {min_priority: 'high', types: ['improvement','bug']}
auto_release BOOLEAN DEFAULT true,
auto_develop BOOLEAN DEFAULT true,
model_id UUID REFERENCES kaizen_ai_models(id),
template_id TEXT,
status TEXT DEFAULT 'active',
last_run_at TIMESTAMPTZ,
created_at TIMESTAMPTZ DEFAULT NOW(),
updated_at TIMESTAMPTZ DEFAULT NOW()
```

#### Цикл работы:

```
[Cron тик]
  → improve (AI генерирует задачи)
  → auto-approve (по правилам: priority >= high)
  → auto-release (собрать утверждённые задачи в релиз)
  → prepare_spec (спецификация)
  → develop_release (разработка)
  → prepare_press_release (пресс-релиз)
  → publish (опубликовать)
  → [ждать следующий тик]
```

## 6. Итоговая оценка

**Сильные стороны:**
- Уникальная комбинация полного цикла (идея → код → PR-материалы)
- 6 AI-провайдеров с управлением параллелизмом
- Планировщик с зависимостями шагов
- Self-hosted, минимум зависимостей (3 npm-пакета)
- Чистая архитектура (ESM, REST, PostgreSQL)

**Что улучшить:**
- Ручное утверждение задач — основное узкое место
- Нет API для полного конвейера одной командой
- Нет MCP-интеграции для Claude Code
- Нет cron-триггеров для автозапуска
- Нет WebSocket для real-time обновлений (HTTP polling 4с/10с)

**Стратегия**: MCP-сервер → тестирование → autopipeline → cron/webhooks

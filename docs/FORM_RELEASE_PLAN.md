# План реализации: Формирование релиза из задач (AI)

> Дата: 2026-03-09
> Статус: НА СОГЛАСОВАНИИ

---

## Цель

ИИ анализирует открытые задачи продукта и предлагает один или несколько релизов — группируя задачи по смыслу, приоритету и технической связности. Пользователь может просмотреть и утвердить предложение, либо включить **авто-утверждение** для полностью автоматического формирования.

---

## Место в конвейере

```
Импорт из RC → Улучшение продукта (AI) → ФОРМИРОВАНИЕ РЕЛИЗА (AI) → Спецификация → Разработка → Публикация
                                          ^^^^^^^^^^^^^^^^^^^^^^^^
                                          НОВАЯ ФУНКЦИЯ
```

**До**: пользователь вручную выбирает задачи при создании релиза.
**После**: ИИ предлагает оптимальную группировку, пользователь утверждает (или авто-утверждение).

---

## Архитектура

### Новый тип AI-процесса: `form_release`

```
POST /api/processes
{
  "product_id": "...",
  "type": "form_release",
  "model_id": "...",
  "prompt": "системный промпт + контекст задач",
  "config": {
    "max_releases": 3,           // макс. количество предлагаемых релизов
    "strategy": "balanced",       // стратегия группировки
    "auto_approve": false         // авто-утверждение
  }
}
```

### Стратегии группировки (`strategy`)

| Стратегия | Описание | Когда использовать |
|-----------|----------|--------------------|
| `critical_first` | Первый релиз — все критичные/высокие баги, остальное — позже | Много багов, нужна стабилизация |
| `by_topic` | Группировка по функциональным областям (UI, API, безопасность...) | Разнородные задачи |
| `balanced` | Сбалансированные релизы по объёму и приоритету | По умолчанию, универсальная |
| `single` | Всё в один релиз | Мало задач (< 8) |

### Формат ответа ИИ

```json
{
  "releases": [
    {
      "version": "2.1.0",
      "name": "Безопасность и авторизация",
      "description": "Доработки системы безопасности: CAPTCHA, мультисессии, капча-сервис",
      "issue_indices": [0, 2, 5],
      "priority": "high",
      "rationale": "Связанные задачи по модулю авторизации, критичная безопасность"
    },
    {
      "version": "2.2.0",
      "name": "Канбан и визуализация",
      "description": "Улучшения канбан-доски и отображения данных",
      "issue_indices": [1, 3, 4],
      "priority": "medium",
      "rationale": "UI-задачи, можно реализовать параллельно с 2.1.0"
    }
  ],
  "unassigned": [6],
  "summary": "7 задач распределены в 2 релиза. 1 задача не включена (низкий приоритет, требует уточнения)."
}
```

## Процесс (пользовательский сценарий)

### Ручной режим (auto_approve = false)

```
[Кнопка «Сформировать релиз (AI)»]
        ↓
[Модал настроек: стратегия, кол-во релизов, модель]
        ↓
[AI-процесс запускается → очередь → выполнение]
        ↓
[Результат: предложение N релизов]
        ↓
[Модал обзора: список релизов с задачами]
  ├── Можно перетащить задачу между релизами
  ├── Можно убрать задачу из релиза
  ├── Можно изменить название/версию
  └── [Утвердить] / [Назад]
        ↓
[Создаются релизы → задачи переходят в in_release]
```

### Автоматический режим (auto_approve = true)

```
[Запуск процесса с auto_approve: true]
        ↓
[AI-процесс выполняется]
        ↓
[Автоматическое создание релизов по предложению ИИ]
        ↓
[Результат в логах процесса: созданные релизы + статистика]
```

## Этапы реализации

### Этап 1: Промпт и парсинг

**Файл**: `server/process-runner.js` — новый case `form_release`

**Системный промпт:**

```
Ты — Product Manager. Проанализируй список задач продукта и предложи
оптимальное распределение по релизам.

Продукт: {product.name}
Описание: {product.description}
Стек: {product.tech_stack}

Стратегия: {strategy}
Максимум релизов: {max_releases}

Задачи (JSON):
{issues_json}

Для каждого релиза укажи:
- version: семантическая версия (инкремент от последнего релиза)
- name: краткое название (2-4 слова)
- description: release notes (что входит, зачем)
- issue_indices: индексы задач из списка (0-based)
- priority: high/medium/low — приоритет релиза
- rationale: почему эти задачи сгруппированы вместе

Правила:
- Критичные баги — в первый релиз
- Связанные по функциональности задачи — вместе
- Размер релиза: 3–10 задач (оптимально 5–7)
- Если задача не подходит ни к одному релизу — в unassigned
- Версию предлагай инкрементально от {last_version}

Ответь строго в JSON без markdown.
```

**Контекст для ИИ** (каждая задача):

```json
{
  "index": 0,
  "title": "...",
  "description": "первые 500 символов...",
  "type": "bug|improvement|feature",
  "priority": "critical|high|medium|low",
  "rc_ticket_id": 1393936025,
  "created_at": "..."
}
```

### Этап 2: Утверждение (approve-releases)

**Новый эндпоинт**: `POST /api/processes/:id/approve-releases`

```javascript
// Тело запроса — скорректированное предложение (или as-is)
{
  "releases": [
    {
      "version": "2.1.0",
      "name": "Безопасность",
      "description": "...",
      "issue_ids": ["uuid1", "uuid2", "uuid3"]
    }
  ]
}
```

**Логика:**
1. Для каждого предложенного релиза → вызвать `releases.create()` с `issue_ids`
2. Задачи → `in_release`
3. Сохранить результат в процессе (`approved_count`, `result`)
4. Вернуть созданные релизы

### Этап 3: Авто-утверждение

В `process-runner.js` после получения ответа ИИ:

```javascript
if (config.auto_approve) {
  // Автоматически создать релизы по предложению ИИ
  const createdReleases = [];
  for (const rel of parsed.releases) {
    const issueIds = rel.issue_indices.map(i => openIssues[i].id);
    const release = await releases.create({
      product_id,
      version: rel.version,
      name: rel.name,
      description: rel.description,
      issue_ids: issueIds
    });
    createdReleases.push(release);
  }
  // Логировать результат
  log(process_id, 'auto_approved', { releases: createdReleases.length, issues: totalIssues });
}
```

### Этап 4: API-эндпоинты

| Метод | Эндпоинт | Описание |
|-------|----------|----------|
| POST | `/api/processes` (type=form_release) | Запустить формирование релиза |
| POST | `/api/processes/:id/approve-releases` | Утвердить предложенные релизы |

> `approve-releases` — новый эндпоинт. Остальное переиспользует существующую инфраструктуру процессов.

### Этап 6: MCP-инструменты

| Tool | Описание |
|------|----------|
| `kaizen_form_release` | Запустить AI-формирование релиза (strategy, max_releases, auto_approve) |

### Этап 7: Интеграция в kaizen_run_pipeline

Расширить `kaizen_run_pipeline` новым шагом:

```
improve → approve → FORM_RELEASE → spec → develop → publish
                    ^^^^^^^^^^^^
                    НОВЫЙ ШАГ (опциональный)
```

Параметр: `form_release: { strategy, max_releases, auto_approve: true }`

Когда включён — после утверждения предложений ИИ, вместо ручного создания релиза, запускается `form_release` с авто-утверждением.

## Ограничения и риски

| Риск | Митигация |
|------|-----------|
| ИИ группирует нелогично | Ручной обзор (по умолчанию), rationale помогает понять логику |
| Мало задач — нет смысла | Минимум 2 задачи для запуска; при < 5 предлагать стратегию `single` |
| Конфликт версий (дубли) | Проверять существующие версии продукта, передавать последнюю в промпт |
| Большой контекст (50+ задач) | Отправлять только title + type + priority + первые 200 символов описания |
| Авто-утверждение создаёт нежелательные релизы | Явный чекбокс + предупреждение в UI; логирование всех действий |
